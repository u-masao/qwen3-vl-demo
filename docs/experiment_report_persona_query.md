# 実験レポート: ペルソナ嗜好ベースクエリ設計による検索タスク難易度調整

**実施日**: 2026-06-10〜11  
**ブランチ**: `main`

---

## 1. 背景

前回実験（`docs/experiment_report_subject_query.md`）では、クエリを主語単語（`"cat"` 等）に変更したところ、ベースモデルが Accuracy@1 = 1.0 / NDCG@10 = 1.0 を達成してしまい FT 効果をまったく測定できないという問題が生じた。

Subject クエリが失敗した根本原因は、「猫の画像を探すクエリに猫の画像を返す」というタスクが Qwen3-VL-Embedding-2B の事前学習で既に完璧に習得されている能力だった点にある。FT で学ぶ余地がなく、評価も FT 効果を示せなかった。

この問題を解消するには、**事前学習モデルが知り得ないような恣意的な対応関係をタスクに埋め込む**必要がある。

---

## 2. 目的

1. 「ペルソナ嗜好ベースクエリ」という新しいタスク設計を実装し、ベースモデル精度をランダムレベルに落とす
2. 上記タスクでの FT 効果（埋め込みモデル）を定量評価する
3. FT 埋め込みモデルとリランカー（base/ft）の組み合わせを 6 パターンで比較する
4. データ拡張（train: 200→500、eval: 50→200）が学習安定性に与える影響を確認する
5. 各処理ステージの計算時間を記録する

---

## 3. 方法

### 3.1 パイプライン構成

```
generate_data → eval_base → train → eval → train_reranker → rerank
```

DVC で依存関係・成果物を管理。ペルソナ列（`persona`）を全ステージに追加したためデータから再生成した。

### 3.2 ペルソナ嗜好ベースタスク設計

タスクの核心アイデア：**視覚的・テキスト的に無関係な subject をペルソナに恣意的に割り当て**、その対応をモデルに学習させる。

| 項目 | 変更前（subject） | 変更後（persona） |
|---|---|---|
| クエリ形式 | `"cat"` | `"user_alpha"` |
| 正例定義 | 同一主語の全画像（視覚的に自明） | 同一ペルソナの全画像（視覚非推測） |
| ベースモデルに解けるか | 完璧に解ける | ほぼ解けない（未知のルール） |

#### ペルソナ設計（7ペルソナ × 5 subjects = 35 subjects）

カテゴリをまたいで非直感的に割り当てることで、視覚・意味の類似性から正解を推測できないようにした。

| ペルソナ | 割り当て subjects |
|---|---|
| user_alpha | cat, pizza, motorcycle, lighthouse, old typewriter |
| user_beta  | dog, burger, bicycle, city street, guitar |
| user_gamma | rabbit, sushi roll, train, forest path, lantern |
| user_delta | horse, coffee cup, sailboat, mountain, ceramic teapot |
| user_epsilon | owl, cupcake, scooter, beach, wooden chair |
| user_zeta  | fox, salad, airplane, desert dune, leather backpack |
| user_eta   | panda, parrot, bowl of ramen, car, waterfall |

例えば `user_alpha` は猫・ピザ・バイクという視覚的に無関係な 5 カテゴリにまたがる。これらの共通点はモデルには（事前学習では）わからない。

### 3.3 マルチポジティブ検索評価

1 クエリに対して複数の正解文書が存在するマルチポジティブ設定を採用した。

```
クエリ: "user_alpha"
正解: user_alpha ペルソナに割り当てられた全画像（猫・ピザ・バイク等）
```

評価指標は NDCG@10 / MRR / Recall@k（マルチポジティブに対応した計算）。

### 3.4 データ生成設定

前回から `num_train`・`num_eval` を拡大し、ペルソナあたりの学習ペア数を増やした。

| 項目 | 設定 |
|---|---|
| 生成モデル | `black-forest-labs/FLUX.2-klein-4B` |
| dtype | bfloat16 |
| 推論ステップ数 | 4 |
| guidance_scale | 1.0 |
| 画像サイズ | 512 × 512 px |
| batch_size | 1（VRAM 節約） |
| train 枚数 | 500（前回の 2.5 倍） |
| eval 枚数 | 200（前回の 4 倍） |

### 3.5 埋め込みモデル FT 設定

| 項目 | 設定 |
|---|---|
| ベースモデル | `Qwen/Qwen3-VL-Embedding-2B` |
| 損失関数 | MultipleNegativesRankingLoss（コサイン類似度） |
| アンカー | ペルソナ名（`"user_alpha"` 等）|
| エポック数 | 1 |
| per_device_batch_size | 4 |
| 勾配累積 | 1 |
| 学習率 | 2.0e-5 |
| warmup_ratio | 0.1 |
| gradient_checkpointing | 有効 |
| 途中評価 | every 50 steps（eval スプリットで NDCG@10 を計測） |

### 3.6 リランカー FT 設定

| 項目 | 設定 |
|---|---|
| ベースモデル | `Qwen/Qwen3-VL-Reranker-2B` |
| 損失関数 | BinaryCrossEntropyLoss |
| クエリ | ペルソナ名 |
| 学習ペア数 | 2000（正例 500 × (1+3)） |
| num_negatives | 3（Hard Negative Mining） |
| エポック数 | 1 |

Hard Negative Mining: FT 済み埋め込みモデルでコーパスをエンコードし、類似度上位の画像（正例を除く）を負例として採用する。

## 4. 結果

### 4.1 計算時間

タイムスタンプから推定した各ステージの所要時間を示す。

| ステージ | 所要時間（推定） | 備考 |
|---|---|---|
| generate_data（train 500枚 + eval 200枚） | 1〜3時間 | FLUX.2-klein-4B、batch_size=1 |
| eval_base | **約 30 秒** | 200件の埋め込み評価 |
| train（埋め込みFT） | **約 17 分** | 500ペア / 1エポック（途中評価 3回 × 2.5分含む） |
| eval（FT後） | **約 2.5 分** | eval_runtime ≈ 165 秒 |
| train_reranker | **推定 1〜2 時間** | HNM + 2000ペア × 1エポック |
| rerank（6パターン評価） | **数分** | embed×2 × rerank×3 |

> **注記**: generate_data の正確な開始時間を記録していないため、所要時間は推定値。前回実験（250枚・約 63 分）との比例から 700 枚は 2〜3 時間程度と推定される。次回以降は `nvidia-smi dmon` 等を併用して計測する。

#### 埋め込みモデル FT 中の損失推移

| ステップ | エポック | 学習損失 | NDCG@10（途中評価） |
|---|---|---|---|
| 10  | 0.08 | 1.617 | — |
| 50  | 0.40 | 0.931 | **0.900** |
| 100 | 0.80 | 0.560 | **0.977** |
| 125 | 1.00 | — | **0.985** |

10 ステップで損失が急減し、step50 時点（epoch 0.4）で既に NDCG@10 = 0.90 を達成。その後も単調に改善した。

### 4.2 埋め込みモデル単体評価（ペルソナクエリ）

| メトリクス | Base | FT後 | 改善幅 |
|---|---|---|---|
| Accuracy@1     | 0.130 | **1.000** | +0.870 |
| Accuracy@3     | 0.255 | **1.000** | +0.745 |
| Accuracy@5     | 0.420 | **1.000** | +0.580 |
| Accuracy@10    | 0.715 | **1.000** | +0.285 |
| Recall@1       | 0.005 | 0.035 | +0.030 |
| Recall@5       | 0.025 | 0.170 | +0.145 |
| Recall@10      | 0.070 | 0.345 | +0.275 |
| NDCG@10        | 0.180 | **0.985** | **+0.805** |
| MRR@10         | 0.247 | **1.000** | **+0.753** |
| MAP@100        | 0.114 | 0.921 | +0.807 |

ベース NDCG@10 = 0.180 はランダムベースライン（7ペルソナなら 1/7 ≈ 0.143）に近い水準であり、タスク難易度設計が意図通りに機能していることを確認できた。

### 4.3 2 段階検索 6 パターン評価（top_k=10）

| パターン | NDCG@1 | NDCG@5 | NDCG@10 | MRR | Recall@10 |
|---|---|---|---|---|---|
| embed=base + rerank=none | 0.130 | 0.137 | 0.180 | 0.247 | 0.070 |
| embed=base + rerank=base | 0.130 | 0.216 | **0.199** | **0.333** | 0.070 |
| embed=base + rerank=ft  | 0.130 | 0.139 | 0.181 | 0.279 | 0.070 |
| embed=ft + rerank=none  | **1.000** | 0.977 | 0.985 | **1.000** | **0.345** |
| embed=ft + rerank=base  | 0.865 | 0.954 | 0.970 | 0.933 | 0.345 |
| embed=ft + rerank=ft    | **1.000** | **1.000** | **0.991** | **1.000** | **0.345** |

### 4.4 前回実験（subject クエリ）との比較

| パターン | NDCG@10（subject） | NDCG@10（persona） | ベース精度差 |
|---|---|---|---|
| embed=base + rerank=none | 1.000 | 0.180 | **-0.820** |
| embed=ft + rerank=none   | 1.000 | 0.985 | -0.015 |
| embed=base + rerank=ft   | 0.976 | 0.181 | — |
| embed=ft + rerank=ft     | 0.981 | **0.991** | — |

ペルソナタスクによりベースモデルの精度を大幅に低下させ、FT 後との差分を明確に測定できるようになった。

## 5. 考察

### 5.1 タスク難易度設計の成功

ベース NDCG@10 = 0.180 はほぼランダムレベルであり、「ペルソナ嗜好ベースの恣意的マッピング」という設計が意図通りに機能した。事前学習済み VLM は `"user_alpha"` と猫・ピザ・バイクの関係を知らないため、クエリと画像を意味的に照合できない。

### 5.2 1 エポック FT での劇的な精度改善

FT 後の NDCG@10 = 0.985 という結果は、500 ペア・1 エポックで**ほぼ完全な暗記**が起きていることを示す。損失推移からも step10〜50 の急減が観察されており、学習データがモデル容量（2B パラメータ）に対して非常に小さいため、過学習に近い収束をしていると考えられる。

train/eval でペルソナが重複している（同一ペルソナのデータを train と eval の両方に持つ）ため、**評価が実質的にトレーニングセットのメンバーシップテスト**になっている可能性が高い。これは未知ペルソナへの汎化能力を測定できていないことを意味する。

### 5.3 FT リランカー × FT 埋め込みの相乗効果

| 組み合わせ | NDCG@10 |
|---|---|
| embed=ft + rerank=none | 0.985 |
| embed=ft + rerank=base | 0.970 |
| embed=ft + rerank=ft   | **0.991** |

FT 済みリランカーは FT 埋め込みと組み合わせると +0.006 の改善をもたらす。一方、base リランカーは FT 埋め込みの結果を -0.015 悪化させる。

これは base リランカーが「ペルソナ名クエリ」という事前学習で見たことのない形式のクエリへの対応が不十分であるためと解釈できる。FT リランカーはペルソナ名クエリで学習されているため、同じ形式での評価では適切にスコアリングできる。

### 5.4 Recall@k が低い理由

Recall@10 = 0.345（FT後）は「200件の eval コーパスから top-10 件の中に正解を何割カバーできるか」を示す。1 ペルソナあたり eval 件数は約 200/7 ≈ 29 件であるのに対し top_k=10 しか取得しないため、理論的な上限は 10/29 ≈ 0.34 前後となる。Recall@10 = 0.345 はこの上限に近く、取得済みの top-10 内には正解が高確率で入っていることを示す（完璧に取得できている）。

### 5.5 データ拡大の効果

前回（train=200）でも精度自体は高かったが、今回（train=500）では損失推移がより安定しており、step50 時点での NDCG@10 が 0.90 と高い水準から始まっている。ペルソナあたりの学習ペア数増加（約 29 → 71）による学習安定化の効果と考えられる。

## 6. 今後の課題

### 6.1 ペルソナ分離（train/eval 分離）— 最優先

現在の設計では同一ペルソナが train と eval の両方に含まれているため、eval が実質的なメモリテストになっている。FT の汎化能力を測定するには、**train には存在しない新しいペルソナを eval に使う**ことが必要である。

```
# 現状
train: user_alpha, beta, gamma, delta, epsilon, zeta, eta
eval:  user_alpha, beta, gamma, delta, epsilon, zeta, eta  ← 同じ

# 改善案
train: user_alpha, beta, gamma, delta, epsilon（5ペルソナ）
eval:  user_zeta, user_eta（2ペルソナ、学習で見ていない）
```

これにより「ペルソナ嗜好パターンの学習能力（新規ペルソナへの外挿）」を測定できる。

### 6.2 クエリの自然言語化

`"user_alpha"` という識別子クエリではなく、ペルソナの嗜好を自然言語で記述したクエリを設計する。

```
# 現状
クエリ: "user_alpha"

# 改善案
クエリ: "I enjoy minimalist aesthetics and warm vintage tones"
```

自然言語クエリは事前学習知識との干渉が生じうるが、より現実的なユースケースに近い。

### 6.3 モデルの過学習防止

現状は 2B パラメータモデルに対して 500 ペアしかなく、1 エポックで NDCG@10 ≈ 1.0 に達してしまう。以下の対策が考えられる：

- **データ量の拡大**: train=2000〜5000 枚にスケールアップし、メモリ容量以上のデータで学習する
- **難化戦略**: 同一ペルソナ内で視覚的に似た画像（同一 subject のバリエーション生成）を増やし、fine-grained 識別を要求する
- **ドロップアウト増加**: 現行の設定ではドロップアウトが最小限であり、過学習を助長している可能性

### 6.4 GPU モニタリングの整備

データ生成ステージの正確な実行時間を把握できていない。次回以降は `nvidia-smi dmon` を並行実行して GPU 使用率・VRAM 消費量・消費電力の推移をログに残す。

### 6.5 バッチサイズの最適化

MNRL の学習品質はバッチサイズに依存する（in-batch negatives の多様性）。現在 per_device_batch_size=4 は VRAM を優先した最小設定であり、gradient_checkpointing を活用してバッチサイズを 8〜16 まで引き上げることで学習品質が改善する可能性がある。
