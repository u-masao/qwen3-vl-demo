# Reranker FT がベースより低かった問題：原因分析と対応記録

**記録日**: 2026-06-09  
**ブランチ**: `claude/nifty-feynman-ZW8gv`

---

## 1. 実験設定

### モデル構成

| 役割 | モデル |
|---|---|
| 埋め込みモデル（ベース） | `Qwen/Qwen3-VL-Embedding-2B` |
| 埋め込みモデル（FT済） | `outputs/model/`（FT後保存先） |
| リランカー（ベース） | `Qwen/Qwen3-VL-Reranker-2B` |
| リランカー（FT済） | `outputs/reranker/`（FT後保存先） |

### データ

- 生成画像数: 250枚（train: 200 / eval: 50）
- 生成モデル: `black-forest-labs/FLUX.2-klein-4B`（4 ステップ推論）
- キャプション形式: `"a fluffy cat on a wooden table"` のような自然言語文

### リランカーFT設定（当初）

| 項目 | 設定 |
|---|---|
| 学習ペア数 | 800（正例200 + 負例600） |
| 負例生成方法 | **ランダムサンプリング**（別カテゴリ優先） |
| num_negatives | 3 |
| エポック数 | 1 |
| 損失関数 | `BinaryCrossEntropyLoss` |

### 評価方法

eval セット 50 件に対して `embed × rerank` の 4 パターンを測定。評価指標: NDCG@10 / MRR / Recall@k。リランク時は top_k=10 の埋め込み候補を取得してリランカーで並べ替え。

---

## 2. 観測されたこと（当初の結果）

| パターン | NDCG@10 | MRR |
|---|---|---|
| embed=base + rerank=base | **0.9852** | 0.9800 |
| embed=base + rerank=ft   | 0.9779 | 0.9700 |
| embed=ft + rerank=base   | **0.9852** | 0.9800 |
| embed=ft + rerank=ft     | 0.9779 | 0.9700 |

**FT リランカーがベースリランカーより NDCG@10 で -0.0073、MRR で -0.0100 低い。**

---

## 3. 検討したこと

### 仮説 A：訓練・評価の分布ミスマッチ（主因と判断）

- **学習時の負例**: ランダムに選んだ別カテゴリ画像（簡単）
- **評価時の候補**: 埋め込みモデルの上位10件（意味的に似た難しい候補）

リランカーが実際に使われる局面（「似ているけど違う」を見極める）と、学習で見ていた負例（「明らかに違う」）の間に大きなギャップがある。これにより評価時には未知の難しさに直面していた。

### 仮説 B：学習データ不足

800 ペアは CrossEncoder の FT としては小規模。汎化性能への影響は否定できないが、主因ではないと判断（データを増やすだけでは仮説 A は解消されない）。

### 仮説 C：過学習

1 エポックかつ小規模データという条件は過学習のリスクを持つ。ただし、評価データとの分布差（仮説 A）の方が説明力が高い。

### 仮説 D：損失関数の評価方法との不一致

BinaryCrossEntropyLoss（2 値分類）でのトレーニングと、推論時の `.rank()` スコア（連続値）の形式差を疑った。ただし、**ベースモデルも同じ `.rank()` で評価しているため**、この差は FT < Base の直接原因にならない。→ 否定。

### 仮説 E（根本原因）：タスクが自然言語的に解ける

クエリが `"a fluffy cat on a wooden table"` のような完全なキャプションの場合、Qwen3-VL が事前学習で習得した text-image 意味マッチング能力で十分に対応できる。FT によって**視覚的識別能力**を新たに学ぶ余地がない。合成データの (caption, image) ペアが 1:1 対応のため、モデルは「このキャプションに含まれる単語と対応する画像を探す」という言語的タスクをこなせばよく、画像間の外観差を学ぶ必要がない。

## 4. 対応内容

### 対応 1：Hard Negative Mining（短期対応）

**変更ファイル**: `src/qwen3vl_demo/train_reranker.py`

FT 済み埋め込みモデルを用いてコーパスをエンコードし、各クエリとのコサイン類似度が高い上位候補（正例を除く）を負例に選ぶ `mine_hard_negatives()` 関数を実装した。

```python
# 類似度上位から自身を除いた num_negatives 件を負例に選ぶ
top_indices = sim[i].topk(fetch_k).indices.tolist()
hard_negs = [j for j in top_indices if j != i][:num_negatives]
```

これにより「評価時にリランカーが受け取る候補の分布」に近い学習環境が実現される。

### 対応 2：Subject ベースのクエリ設計（根本対応）

**変更ファイル**: `prompts.py`, `generate_data.py`, `evaluate.py`, `train.py`, `train_reranker.py`, `rerank.py`

クエリを全キャプション（`"a fluffy cat on a wooden table"`）から**主語単語のみ**（`"cat"`）に変更。正解定義も「同一主語の全画像」に変更した。

| 変更前 | 変更後 |
|---|---|
| クエリ = 全キャプション文 | クエリ = 主語単語（"cat" など） |
| 正解 = 自分のキャプションから生成した画像（1件） | 正解 = 同一主語の全画像（複数件） |

この変更により：
- 「自然言語意味マッチング」タスク → 「視覚的カテゴリ識別」タスク
- 事前学習済みの text-image 対応知識だけでは解けなくなる
- FT で視覚特徴を学ぶ余地が生まれる

## 5. 対応結果

### Hard Negative Mining 後の結果

| パターン | NDCG@10（変更前） | NDCG@10（変更後） | 差分 |
|---|---|---|---|
| embed=base + rerank=base | 0.9852 | 0.9852 | ±0.0000 |
| embed=base + rerank=ft   | 0.9779 | **0.9926** | **+0.0147** |
| embed=ft + rerank=base   | 0.9852 | 0.9852 | ±0.0000 |
| embed=ft + rerank=ft     | 0.9779 | **0.9926** | **+0.0147** |

Hard Negative Mining により FT リランカーがベースリランカーを上回った（NDCG@10: 0.9852 → 0.9926）。

### Subject ベースクエリの結果

Subject クエリでは Accuracy@1 = 1.0 / NDCG@10 = 1.0 とベースモデルが完璧に解けてしまい、
FT による改善余地がゼロになった（`docs/experiments/experiment_report_subject_query.md` 参照）。

その後、さらに難度を上げるためにペルソナ嗜好ベースクエリ設計へ移行した。

### ペルソナ嗜好ベースクエリの結果

| パターン | NDCG@10 | MRR |
|---|---|---|
| embed=base + rerank=none | 0.1804 | 0.2468 |
| embed=base + rerank=base | **0.1992** | **0.3332** |
| embed=base + rerank=ft   | 0.1812 | 0.2788 |
| embed=ft + rerank=none   | 0.9851 | 1.0000 |
| embed=ft + rerank=base   | 0.9703 | 0.9325 |
| embed=ft + rerank=ft     | **0.9911** | **1.0000** |

- FT リランカー（embed=ft+rerank=ft）が FT 埋め込み単体（embed=ft+rerank=none）を上回った
- base 埋め込みでは base リランカーが rerank=none より改善（+0.019 NDCG@10）
- FT 埋め込みに base リランカーを組み合わせると逆効果（-0.015）— ペルソナ名クエリを学習していないため

（詳細は `docs/experiments/experiment_report_persona_query.md` 参照）

## 6. 教訓

1. **評価時の候補分布を学習時に再現する**: リランカーは常に「埋め込みモデルの上位候補」に対して使われる。学習時の負例もこの分布に合わせることが重要。ランダム負例は簡単すぎて評価環境との乖離が生じる。

2. **事前学習済みの能力と FT の役割を区別する**: 事前学習で習得済みのタスク（text-image 意味マッチング）を FT で繰り返しても改善幅は小さい。FT 固有の貢献が生まれるタスク設計（視覚分類、ドメイン固有パターン等）を意識する。

3. **評価方法の一貫性を先に確認する**: FT < Base の比較では「評価方法が同一か」を最初に確認することで、見当違いな原因調査を省ける。
