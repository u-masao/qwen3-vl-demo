# 実験レポート: Subject ベースクエリ設計による視覚分類タスクへの変更

**実施日**: 2026-06-09〜10  
**ブランチ**: `claude/nifty-feynman-ZW8gv`

---

## 1. 背景

前回実験（`docs/experiment_report.md`）では、クエリをキャプション全文（`"a fluffy cat on a wooden table"`）として合成データで埋め込みモデルをFTしたところ、Accuracy@1 が 86% → 92% に改善した。しかしリランカーFTはベースより -0.0074 低い結果となった。

この問題を分析した結果、**根本的なタスク設計の問題**が疑われた:

- クエリが自然言語キャプション文のとき、Qwen3-VL-Embedding-2B が事前学習で習得済みの text-image 意味マッチング能力だけで十分対応できる
- FTで新たに学ぶのは「同じキャプションと同じ画像の対応」という同じ種類の能力であり、視覚的識別能力への寄与がない
- 負例も簡単（ランダム選択）であり、評価時の「上位 k 件の埋め込み候補」と分布が乖離していた

これを解消するため、以下の 2 つの変更を組み合わせた実験を実施した:

1. **Subject ベースクエリ設計**: クエリを主語単語のみ（`"cat"`）に変更し、同一主語の全画像を正例とする
2. **Hard Negative Mining**: FT 済み埋め込みモデルで類似度上位の候補を負例に使う

---

## 2. 目的

1. Subject ベースクエリ（`"cat"` など）を用いた視覚分類タスクに変更することで、FT が視覚的識別能力を獲得できるか検証する
2. Hard Negative Mining がリランカーFTの精度改善に寄与するか検証する
3. 変更後のベースモデル精度・FT効果・リランカー効果を定量評価する
4. 各処理ステージの計算時間を計測し、パイプライン全体のコストを把握する

---

## 3. 方法

### 3.1 パイプライン構成

```
generate_data → eval_base → train → eval → train_reranker → rerank
```

DVC で再現性を管理。全ステージを `generate_data` から再実行（subject カラム追加によりスキーマ変更が発生）。

### 3.2 Subject ベースクエリ設計

| 項目 | 変更前 | 変更後 |
|---|---|---|
| クエリ形式 | `"a fluffy cat on a wooden table"` | `"cat"` |
| 正例定義 | 同一キャプションから生成した画像（1件） | 同一主語の全画像（1〜n件） |
| 評価の性質 | テキスト→画像の意味マッチング | 視覚的カテゴリ識別 |

`prompts.py` の `Sample` dataclass に `subject` フィールドを追加し、データセット・評価器・学習・リランク評価の全工程で `subject` をクエリとして使用するよう変更した。

### 3.3 Hard Negative Mining

リランカーFT の負例生成を「ランダムサンプリング」から「埋め込みモデルの上位候補」に変更した。

```python
# FT 済み埋め込みモデルでコーパスをエンコードし、
# 各クエリの類似度上位（正例を除く）を負例に選ぶ
top_indices = sim[i].topk(num_negatives + 1).indices.tolist()
hard_negs = [j for j in top_indices if j != i][:num_negatives]
```

### 3.4 データ生成設定

| 項目 | 設定 |
|---|---|
| 生成モデル | `black-forest-labs/FLUX.2-klein-4B` |
| dtype | bfloat16 |
| 推論ステップ数 | 4 |
| guidance_scale | 1.0 |
| 画像サイズ | 512 × 512 px |
| batch_size | 1（VRAM 節約） |
| train / eval | 200 枚 / 50 枚 |

### 3.5 埋め込みモデルFT設定

| 項目 | 設定 |
|---|---|
| ベースモデル | `Qwen/Qwen3-VL-Embedding-2B` |
| 損失関数 | MultipleNegativesRankingLoss（コサイン類似度） |
| エポック数 | 1 |
| per_device_batch_size | 4 |
| 学習率 | 2.0e-5 |
| warmup_ratio | 0.1 |
| gradient_checkpointing | 有効 |

### 3.6 リランカーFT設定

| 項目 | 設定 |
|---|---|
| ベースモデル | `Qwen/Qwen3-VL-Reranker-2B` |
| 損失関数 | BinaryCrossEntropyLoss |
| 学習ペア数 | 800（正例200 × (1+3)） |
| num_negatives | 3（Hard Negative Mining） |
| エポック数 | 1 |

## 5. 考察

### 5.1 ベースモデルが subject クエリを完璧に処理できる

NDCG@10 = 1.0、MRR = 1.0 という結果は、Qwen3-VL-Embedding-2B が `"cat"` → 猫画像の対応を**事前学習だけで完全に習得済み**であることを示す。subject 単語クエリは人間にとっても画像検索の基礎的な問い（「猫の画像を探す」）であり、大規模 VLM の事前学習データにこの種のペアが豊富に含まれていると考えられる。

### 5.2 FT の改善余地がゼロ

ベースで NDCG@10 = 1.0 を達成してしまっているため、FT による改善余地がない。embed=base と embed=ft が完全に同スコアとなり、**4分8秒の学習コストに対してリターンがゼロ**という結果になった。これは FT の効果を測定するための評価タスクが易しすぎることを意味する。

### 5.3 リランカーが逆効果

rerank=none（NDCG@10=1.0）に対して rerank=base（0.9965）・rerank=ft（0.9761〜0.9810）はいずれも性能が低下した。原因として:

- **クエリ形式の不一致**: リランカーは学習時に `"a fluffy cat on a wooden table"` のような長文クエリで学習されており、`"cat"` のような単語クエリへの対応が想定外
- **リランク対象の問題**: 埋め込み検索が既に完璧な順位を返しているため、リランカーが介入するほど悪化する

### 5.4 タスク設計とモデル能力の整合性

今回の実験で明らかになったのは「タスク難易度とモデル能力のバランス」の重要性である:

| クエリ形式 | タスク難易度 | ベース精度 | FT の効果 |
|---|---|---|---|
| キャプション全文 | 中程度 | NDCG=0.94 | +0.03〜+0.06 |
| 主語単語のみ | 簡単すぎ | NDCG=1.00 | ゼロ |
| 理想（未実施） | 適度に困難 | NDCG < 0.90 | 明確な改善 |

FT の効果が現れるには、ベースモデルが解けない程度の難しさが必要である。

### 5.5 Hard Negative Mining の評価

前回実験（caption クエリ）では Hard Negative Mining により rerank=ft が rerank=base を上回る改善（NDCG: 0.9779 → 0.9926）が確認できていた。今回の subject クエリでは上述の通りリランカー自体が逆効果となっているため、Hard Negative Mining の効果を切り離して評価できていない。

### 5.6 eval 生成速度の著しい低下

train 200枚の生成速度（0.18枚/秒）に対して eval 50枚は 0.019枚/秒 と約 10 倍遅い結果となった。今回の実験では生成中の GPU 使用率・VRAM 消費量・システム CPU 負荷を継続的に計測していないため、原因は特定できない。再現性の確認と原因調査のためには、次回の実験から GPU モニタリング（`nvidia-smi dmon` 等）を並行実行することが望ましい。

## 6. 今後の課題

### 6.1 タスク難易度の調整（最優先）

ベースモデルが解けない程度のタスク設計が必要。いくつかの方向性:

**案A: 同一主語内の fine-grained 識別**  
同じ `"cat"` でも「毛色・姿勢・背景が異なる複数の猫画像」の中から特定の一枚を識別させる。現在のデータは各主語の画像数が少なく（平均 1.72 枚/主語）、fine-grained 識別になっていない。生成枚数を増やす（例: 1000枚）か、生成バリエーションを意図的に増やす（同一主語 × 複数形容詞/情景）ことで実現できる。

**案B: ドメイン固有の主語**  
一般的な「cat」ではなく、専門ドメインの固有名詞（商品型番、医療用語、建築様式名など）を使う。事前学習データに少ない概念を使うことでベースの精度を下げ、FT 効果を測定しやすくする。

**案C: クエリに意図的なノイズを加える**  
主語単語に誤字・類義語置換・言語切り替え（日本語クエリ）などを加え、ベースモデルが苦手とする形式にする。

### 6.2 リランカーのクエリ対応改善

subject 単語クエリへの対応をリランカーに学習させる。Hard Negative Mining とのセットで:
- `(subject_word, image)` ペアで学習
- 正例: 同一主語の画像（複数）
- 負例: Hard Negative Mining による類似主語画像

### 6.3 GPU モニタリングの整備

- `nvidia-smi dmon` を DVC 実行と並行して記録し、VRAM 使用量・GPU 使用率・消費電力の推移をログに残す
- eval 生成速度低下の原因を特定するためにも、次回実験からモニタリングを標準化する
- batch_size チューニング: 現在 batch_size=1 でシリアル処理。GPU 負荷の実測値をもとに最適バッチサイズを探索する

### 6.4 データ量の拡大

200枚という小規模データセットでは FT の効果が小さく出やすい。`num_train` を 1000〜5000 枚に増やし、FT のスケーリング効果を検証する。評価セットも 50 枚では統計的信頼性が低く、200〜500 枚が望ましい。

### 6.5 Flash Attention 2 の導入

現在 sdpa にフォールバック中。`flash-attn` のインストールにより埋め込みモデルの学習・推論を高速化できる。generate_data のボトルネックは拡散モデル推論であり直接影響しないが、train/eval ステージの時間短縮が期待できる。
